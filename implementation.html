System Architecture

The diagram becomes an interactive architecture reader.

System Layers And User Entry Points

The system adopts a frontend-only static multi-page architecture that organises the application into clearly defined layers: User, Presentation, Logic, Data/State, and External Services/Deployment. Two primary user groups, visitors and residents, access the system through five main interfaces: Homepage, Interactive Map, Photo Wall, Notice Board, and Smart Agent.

You can click on any layer to view the detailed introduction of that layer.

Implementation Summary

Overall, this architecture provides a lightweight yet functional implementation model, enabling interactive user experiences within a static deployment environment while maintaining a clear separation between interface, logic, data, and service dependencies.

Selected Layer

Presentation Layer

At the presentation level, user interactions such as tapping, browsing, submitting, asking, and viewing are captured and translated into page events.

Selected Layer

Data And State Layer

At the data level, the current prototype relies on embedded content datasets, static assets, browser file previews, localStorage preferences, and UI/interaction states, rather than a backend server or database. As a result, most system behaviour is executed directly in the browser: scripts read embedded content, update local interface states, retrieve selected external data, and render feedback back to users in the form of cards, modals, previews, and dynamic text updates.

Selected Layer

External Services

External integrations are used selectively to extend functionality, including Open-Meteo APIs for weather information, AMap SDK for live map rendering and hotspot markers, and Google Fonts for visual presentation.

Selected Layer

Logic Layer

These events are then processed by the logic layer, which combines a shared controller, script.js, with page-specific scripts including interactive-map.js, photo-wall.js, notice-board.js, and smart-agent.js. This layered structure ensures that shared interface behaviours are managed consistently while allowing each page to support its own specialised interactions.
